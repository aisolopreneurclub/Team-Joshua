# 오픈클로 셋업 가이드

## 구조

```
오픈클로 워크스페이스 2개:

봇 1: 준호 (개발자)          봇 2: 민재 (팀장)
├── SOUL.md (준호용)         ├── SOUL.md (민재용)
├── AGENTS.md (기존)         ├── AGENTS.md (기존)
├── USER.md (기존)           ├── USER.md (기존)
└── TOOLS.md (기존)          └── TOOLS.md (기존)
```

## 셋업 순서

### 1. 준호 봇 (개발자)

1. 오픈클로에서 새 워크스페이스 생성 (또는 기존 봇에 퍼소나 추가)
2. `openclaw-junho-SOUL.md`의 내용을 해당 봇의 `SOUL.md`에 붙여넣기
3. 이름/아바타 설정: "준호" / 개발자 아이콘

### 2. 민재 봇 (팀장)

1. 오픈클로에서 새 워크스페이스 생성 (또는 기존 봇에 퍼소나 추가)
2. `openclaw-minjae-SOUL.md`의 내용을 해당 봇의 `SOUL.md`에 붙여넣기
3. 이름/아바타 설정: "민재" / 팀장 아이콘

### 3. 설계서 전달 (1회)

Slack 채널이나 오픈클로 대화에서:

```
@준호 @민재

ERP 프로젝트 설계서입니다. 이 문서를 기반으로 작업합니다.

[internal-erp.design.md 내용 붙여넣기]
```

### 4. 작업 시작

```
@준호 Cycle 2 작업 시작해.

[cycle-2-detailed.md 내용 붙여넣기]
```

준호가 구현 완료 보고하면:

```
@민재 준호가 Cycle 2 구현했어. 리뷰해줘.
```

민재가 리뷰 결과를 보내면:

- PASS → 다음 사이클 지시
- FAIL → 준호가 자동으로 수정 진행 (`@준호` 멘션 불필요, FAIL 내용 보고 자동 수정)

## 티키타카 흐름

```
Josh(@집사): @준호 Cycle 2 시작해. [지시서 붙여넣기]
          ↓
준호: [구현] → [완료 보고] @민재 리뷰 부탁드립니다.
          ↓
민재: [리뷰] → FAIL: 수정 3건. @준호 수정해 주세요.
          ↓
준호: [수정] → [재보고] @민재 수정 완료했습니다.
          ↓
민재: [재리뷰] → PASS. 다음 Cycle 진행 가능.
          ↓
Josh(@집사): @준호 Cycle 3 시작해. [지시서 붙여넣기]
```

## 사람 개입 포인트

| 시점 | 사람이 할 일 |
|------|-------------|
| 사이클 시작 | 지시서 붙여넣기 + @준호 멘션 |
| PASS 후 | 다음 사이클 지시서 투입 |
| ESCALATE 시 | 판단 후 방향 결정 |
| 전체 완료 | 최종 코드 확인 + 머지 |

## 비용 최적화

- 설계서는 **1회만** 전달 (컨텍스트에 남음)
- 상세 지시서(cycle-N-detailed.md)를 사용하면 준호가 생각할 양 감소
- FAIL 시 민재가 7건 이하로 피드백 → 수정 1회로 PASS 가능성 높음
- 사이클 간 대화가 너무 길어지면 새 스레드 시작 (토큰 리셋)
